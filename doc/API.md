# ShagunBeautyBe — Backend API Reference

This document describes every HTTP endpoint exposed by the `theceliacstore-BE` Express/MongoDB backend, as implemented in the code (routes + controllers) at the time of writing. All paths are relative to the server root, e.g. `http://localhost:5000` in local development (see `env.example`, `APP_URL`). In production this is whatever the deployed host is.

This document was generated by reading `server.js`, every file under `routes/`, every file under `controllers/`, the `middleware/auth/*` guards, and the relevant Mongoose models/services where needed to determine response shapes. No Swagger/JSDoc `@swagger` annotations exist anywhere in `routes/**/*.js` (verified by search), so `swagger.js`/`/api-doc` currently serves an effectively empty spec (just `info` + `components.securitySchemes.bearerAuth`) — it is not a source of truth for any endpoint.

## General Notes

- **Base URL**: All routes below are mounted under `/api/...` as shown in `server.js`. Example: `GET /api/product` → `http://localhost:5000/api/product`.
- **Content-Type**: Send `application/json` for all JSON bodies. Endpoints that accept file uploads (see each module) require `multipart/form-data` and use [multer](https://www.npmjs.com/package/multer) disk storage (files land under `uploads/images|voice_notes|pdfs` before being pushed to Cloudinary — see `config/multer.js` and `utils/upload/index.js`).
- **Body size limit**: `express.json({ limit: "50mb" })` and `express.urlencoded({ limit: "50mb", extended: true })` — both JSON and urlencoded bodies are capped at 50 MB (`server.js`).
- **CORS**: `cors({ origin: "*", credentials: true })` — all origins are allowed. Note this combination is unusual (browsers will not honor `credentials: true` with a wildcard origin in practice), but that's what's configured.
- **Cookies**: `cookie-parser` is loaded, but no route in this codebase actually reads/writes an auth cookie — authentication is exclusively via Bearer JWTs (see below). No endpoint sets a session cookie.
- **Auth scheme**: Two independent JWT-based identity systems, both HS256 JWTs signed with `process.env.JWT_SECRET`, sent as `Authorization: Bearer <token>`:
  - **Admin/Super-Admin/Sub-Admin** tokens — validated by `middleware/auth/adminMiddleware.js`. Looks up an `Admin` document by the token's `id` claim, then checks `admin.role` against an allow-list. Exposes `req.admin`. Guards used across routes: `superAdmin` (role `super_admin` only), `admin` (role `admin` only), `subAdmin` (role `sub_admin` only), `adminOrSubAdmin` (`admin`/`sub_admin`), `adminOrSuperAdmin` (`admin`/`super_admin`), `adminOrSubAdminOrSuperAdmin` (all three roles).
  - **User** tokens — validated by `middleware/auth/userMiddleware.js`. Looks up a `User` document by the token's `id` claim (no role concept). Exposes `req.user`.
  - Both middlewares: return `401 { data: null, message: "Unauthorized", success: false }` if no `Authorization` header/token is present; return `403 { data: null, message: "Access denied", success: false }` if the principal isn't found or (for admin) doesn't have an allowed role; return `403 { data: null, message: "Invalid or expired token", success: false }` if `jwt.verify` throws (bad signature/expired/malformed).
  - Tokens are generated via `utils/auth` (`generateAccessToken`) at register/login time and returned in the response body as `token` — there is no refresh-token flow wired up (refresh token generation is commented out in the admin/user auth controllers).
- **Standard success envelope** — the vast majority of endpoints respond via `new ApiResponse(statusCode, data, message, success)` (`utils/ApiResponse.js`), which serializes as:
  ```json
  {
    "statusCode": 200,
    "data": { "...": "endpoint-specific payload" },
    "message": "Human readable message",
    "success": true
  }
  ```
  A handful of endpoints (noted inline) bypass `ApiResponse` and return a raw object, e.g. file-export endpoints that just return `{ "url": "..." }`, or the email-tracking pixel/redirect endpoints which return binary/redirect responses.
- **Standard error envelope** — uncaught exceptions inside any route are caught by `common/asyncHandler.js`, which responds:
  ```json
  { "success": false, "message": "<error.message>" }
  ```
  with status `error.code || 500` (in practice `error.code` is rarely set by application code, so unhandled exceptions almost always surface as plain `500`). Most *expected* error paths (validation failures, not-found, etc.) are handled explicitly inside each controller using `ApiResponse` instead — those are documented per-endpoint below. Note that many controllers call `res.json(new ApiResponse(400/404/..., ...))` **without** also calling `.status(...)`, meaning the HTTP status code actually returned is `200` even though the JSON body's `statusCode` field says otherwise. Where this happens it is called out per endpoint ("HTTP status stays 200" ) — this is a real quirk of the codebase, not a documentation error.
- **IDs**: All `:id` style path params are MongoDB ObjectIds; most controllers validate them with `mongoose.Types.ObjectId.isValid()` and return a `400` (via `ApiResponse`, HTTP status usually still `200` — see above) for malformed ids.
- **Pagination convention**: List endpoints generally accept `page` (1-based) and `per_page` query params and return either `{ data: [...], total: N }` or `{ categories/brands/zones: [...], total, page, per_page, total_pages }` — the exact shape is noted per endpoint/module since it isn't fully consistent across modules.
- **Decimal fields**: `price`, `discounted_price`, order totals, etc. are stored as Mongoose `Decimal128` but converted to plain JS numbers in `toJSON` transforms / controller code before being sent to clients — response examples below show them as plain numbers.

## Auth — Admin

Base path: `/api/auth/admin`. Source: `routes/auth/admin/index.js`, `controllers/auth/admin/index.js`, model `models/adminModel.js`.

Admin schema fields: `name` (String, required), `email` (String, required, unique), `password` (String, required, bcrypt-hashed on save), `role` (enum `super_admin|admin|sub_admin`, default `admin`), `services` ([ObjectId ref Services]), `created_by` (ObjectId ref Admin, required only when `role === "sub_admin"`).

### GET /api/auth/admin/

List all admins with `role: "admin"` (paginated, searchable). Super-admin management screen.

**Auth**: `Authorization: Bearer <admin JWT>` — `superAdmin` only.

**Query params**

| Param | Type | Required | Description |
|---|---|---|---|
| search | string | No | Case-insensitive match on `name` or `email` |
| page | number | No | Default `1` |
| per_page | number | No | Default `50` |

**Request body**: None.

**Success response** `200`:
```json
{
  "statusCode": 200,
  "data": {
    "data": [
      { "_id": "...", "name": "Admin One", "email": "admin1@store.com", "role": "admin", "created_by": null, "createdAt": "...", "updatedAt": "..." }
    ],
    "total": 1
  },
  "message": "Admins fetched successfully",
  "success": true
}
```

**Errors**:
- `req.admin._id` missing (should not happen, guarded by middleware) → body `{"statusCode":404,"data":null,"message":"Not authorized"}`, HTTP status stays `200`.

**Errors**:
- Email already exists → `{"statusCode":400,"data":null,"message":"Admin already exists","success":false}` (HTTP 400 via `.status(400)`).

Fetch one admin by id.

**Auth**: `Authorization: Bearer <admin JWT>` — `superAdmin` only.

**Path params**: `id` (ObjectId, required).

## Auth — User

Base path: `/api/auth/user`. Source: `routes/auth/user/index.js`, `controllers/auth/user/index.js`, model `models/userModel.js`.

User schema: `name` (required), `email` (required, unique), `password` (optional — absent for Google users), `phone` (optional, unique, sparse), `googleId` (unique, sparse), `authProvider` (enum `local|google`, default `local`), `profilePicture`.

### GET /api/auth/user/

List all users (paginated, searchable). Despite living under `/api/auth/user`, this is guarded by the **admin** middleware, not user.

**Auth**: `Authorization: Bearer <admin JWT>` — `superAdmin` only.

**Query params**: `search` (string, optional — matches both `name` AND `email` simultaneously via a single `$and`-style object, so in practice this filter behaves like "name regex AND email regex" rather than OR — likely a bug but documented as coded), `page` (number, optional), `per_page` (number, default 50).

**Success response** `200`: `{ "data": [ { "_id":"...", "name":"...", "email":"...", "phone":"...", "authProvider":"local" } ], "message": "Users fetched successfully" }` (password excluded).

**Errors**: `req.admin._id` missing → `{"statusCode":404,"message":"Not authorized"}`.

---

### POST /api/auth/user/register

Register a new customer account (email/password).

**Auth**: None.

**Request body**:
```json
{ "name": "string (required)", "email": "string (required)", "phone": "string (required — checked for uniqueness)", "password": "string (required)" }
```

**Success response** `200` (body `statusCode: 201`):
```json
{ "data": { "id": "...", "name": "...", "email": "...", "token": "<jwt>" }, "message": "New user created successfully", "success": true }
```

**Errors**:
- `400 {"message":"User already exists"}` — email taken.
- `400 {"message":"Phone number already exists"}` — phone taken.

**Side effects**: Sends a welcome email asynchronously (`sendWelcomeEmail`, fire-and-forget via `setImmediate`, does not block/affect the response; failures are only logged).

**Errors**: `401 {"message":"Invalid credentials"}` (no user found, or user has no password i.e. Google-only account, or password mismatch).

### POST /api/auth/user/google

Google OAuth login/registration. Verifies a Google ID token server-side and creates or links a user account.

**Auth**: None (this route itself is the login mechanism).

**Request body**: `{ "idToken": "string (required — Google ID token from client-side Google Sign-In)" }`

**Success response** `200`:
```json
{
  "data": {
    "id": "...", "name": "...", "email": "...", "phone": null,
    "profilePicture": "https://...", "authProvider": "google", "token": "<jwt>"
  },
  "message": "Google login successful", "success": true
}
```

**Errors**:
- `400 {"message":"Google ID token is required"}`
- `400 {"message":"Email not provided by Google"}`
- `401 {"message":"Invalid or expired Google token"}` — token verification failed with "Token used too late"/"Invalid token"
- `500 {"message": "<error.message or 'Google authentication failed'>"}` — other verification errors

**Notable behavior**: Looks up user by `googleId` first, then by `email` (to link an existing local account to Google — sets `googleId`, `authProvider`, optionally `profilePicture`). If neither exists, creates a brand-new user with no `phone` field at all (deliberately omitted to avoid unique-index collisions between multiple Google users), and fires a welcome email asynchronously.

### POST /api/auth/user/forgot-password

Resets the user's password to a new random value and emails it to them (no reset-link/token flow — this is an immediate reset-and-email).

**Auth**: None.

**Request body**: `{ "email": "string (required)" }`

**Success response** `200`: `{ "data": null, "message": "Password reset successful! Check your email for the new password.", "success": true }`

**Errors**:
- `400 {"message":"Email is required"}`
- `404 {"message":"User not found with this email"}`

**Notable behavior**: Generates an 8-ish char random password (`crypto.randomBytes(4).toString("hex") + <random uppercase letter> + "!"`), saves it (hashed by the pre-save hook), and asynchronously emails the **plaintext** new password to the user via `sendForgotPasswordEmail` (fire-and-forget, failures only logged).

## Category

Base path: `/api/category`. Source: `routes/category/index.js`, `controllers/category/index.js`, model `models/categoryModel.js`.

Category schema: `name` (String, required, max 255), `description` (String, max 2056), `discount_label_text` (String, max 255), `meta_data` (Map, default `{}`), `newly_launched` (Boolean, default false), `is_active` (Boolean, default true), `images` ([String], default `[]`).

### POST /api/category/

Create a category. **Multipart upload** — `multer` `.array("images")`, any number of image files under field name `images`.

**Auth**: `Authorization: Bearer <admin JWT>` — `adminOrSuperAdmin`.

**Headers**: `Content-Type: multipart/form-data`.

**Request body** (form-data fields, alongside the `images` file(s)):
```json
{
  "name": "string (required)",
  "description": "string (optional)",
  "discount_label_text": "string (optional)",
  "meta_data": "JSON string, e.g. '{\"key\":\"value\"}' (optional — parsed server-side)",
  "newly_launched": "'true'|'false' (optional)",
  "is_active": "'true'|'false' (optional)"
}
```

**Success response** `200` (body `statusCode: 201`):
```json
{ "data": { "_id":"...", "name":"Snacks", "images":["https://res.cloudinary.com/..."], "is_active": true, "createdAt":"..." }, "message": "Category created successfully", "success": true }
```

**Errors**:
- No files at all in the request → `404 {"message":"No Image Found"}` (note: checks `req.files` truthiness, which with multer is always an array even if empty, so this only triggers if multer itself failed to attach `req.files`).

## Sub-Category

Base path: `/api/sub-category`. Source: `routes/sub-category/index.js`, `controllers/sub-category/index.js`, model `models/subCategoryModel.js`.

### GET /api/email-tracking/pixel/:orderId/:emailType

Tracking pixel — records that an email was opened, then returns a 1×1 transparent GIF regardless of outcome.

**Auth**: None.

**Path params**: `orderId` (ObjectId, required — invalid ids are tolerated, just skip the DB update), `emailType` (string, required — literal `"confirmation"` or `"status-<status>"`, e.g. `"status-shipped"`).

## Webhook

Base path: `/api/webhook`. Source: `routes/webhook/index.js`, handler implemented in `controllers/order/index.js` (`handlePaymentWebhook`).

### POST /api/webhook/razorpay/payment

Razorpay payment webhook receiver — updates order payment status when Razorpay reports a payment event. Intended to be called by Razorpay's servers, not by application clients.

**Auth**: None via Express middleware (deliberately, since Razorpay itself calls this). Instead, authenticity is verified via **HMAC signature**: the `x-razorpay-signature` request header must equal `HMAC-SHA256(JSON.stringify(req.body), process.env.RAZORPAY_WEBHOOK_SECRET)` in hex.

**Headers**

| Header | Required | Description |
|---|---|---|
| x-razorpay-signature | Yes | HMAC-SHA256 signature of the raw JSON body, computed with `RAZORPAY_WEBHOOK_SECRET` |

**Request body**: Razorpay webhook event payload (JSON) — shape defined by Razorpay, notably `event` (e.g. `"payment.captured"`, `"payment.failed"`) and `payload.payment.entity` / `payload.payment_link.entity`.

**Success response** `200`: `{ "status": "success" }` (not wrapped in `ApiResponse`) — returned even if the event type is unhandled or the referenced order can't be found (those cases are only logged, not surfaced as errors, since Razorpay expects a `200` to stop retrying).

**Errors**:
- `400 {"error":"Invalid signature"}` — signature mismatch.
- `500 {"error":"Webhook processing failed"}` — unexpected exception.

**Notable behavior**: On `payment.captured`, looks up the order by `paymentLinkId`, verifies the paid amount (in paise) matches `order.finalTotalAmount`, and if so sets `paymentStatus: "paid"`, `paymentId`, `paymentMethod`, `paidAt`, and bumps `status` from `"pending"` to `"confirmed"`. On `payment.failed`, sets `paymentStatus: "failed"`. **Important caveat**: this route computes the HMAC over `JSON.stringify(req.body)`, i.e. the body *after* Express's `express.json()` middleware has already parsed and re-serialized it — if Razorpay's actual raw request bytes differ in formatting/key order from Node's re-serialization, signature verification could fail even for genuine requests; a raw-body capture is generally recommended for webhook signature verification and does not appear to be used here.
